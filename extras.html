USER FLOWS

User flows were very helpful for the whole team to understand requirements.

Why: Requirements used to change very quickly whilst developing the MVP so was very important to update the user flows with the current state.

Evolution: Gathering the whole team we would go through all the user flow and change it if we agreed on it. After those meetings the design team could follow the user flow to generate wireframes. They were also very helpful for the developers, product owners, testers..

USER FLOW AXURE TEAM DISCUSSION MVP

WIREFRAMES

Since the first requirements we started wireframing in Axure what it could be the portal look.

Why: Although we knew the look would end up being different we started putting together all requirements folowing the user flow and do a interactive wireframe

Evolution: Team discussions, usability testing, different copy.. the interactive wireframes made in Axure allowed us to quickly update them.

AXURE WIREFRAME INTERACTIVE

VISUAL DESIGN

By first sketching some ideas in Illustrator and Photoshop, we started coding and designing directly in the browser.

Why: In the design team we coded all the front end so designing in the browser since an early stage was easy for us and allowed us to quickly test and update the code.

Evolution: The UI continually changed based on regular user testing feedback. Because we were working on the final code was very easy to make changes and test again. We also created a style guide that was changing at the same time as the project.

END OF PROJECT

Although we had a tight deadline we managed to deliver an MVP fully working on time.

How: By following agile and lean methodologies, pair programing and pair designing, doing internal and guerrilla usability testing.

Awards: Thunderstorm just won an award as the most innovative product in Cinemacon 2016, Las Vegas.

IDEAS MVP

At first we had a very complicated idea of the app, so we decided to begin with an MVP.

Why: An app full of features would have made us loose a lot of time without testing the market first.

Evolution: We started thinking what would be the minimum for the app to be interesting, we researched competitors, sketched first ideas, think who would be our users and refined after that.

IDEAS RESEARCH SKETCHING MVP PERSONAS

WIREFRAMES

Whilst thinking about the MVP we started sketching wireframes, page flows, even putting together some visual elements.

Why: To have an idea of what we wanted to achieve and what could be the future problems.

Evolution: The design evolved with the ideas and the developer was able to start developing as soon as we thought we were having an MVP product in mind.

ILLUSTRATOR WIREFRAME PAGE FLOWS

USABILITY TESTING

The visual design evolved based on usability testing feedback.

Why: The first look and feel was created based on assumptions, we knew that after testing it, we would find some usability problems.

Evolution: Guerrilla usability testing using friends was our approach as they matched our user personas description and we didn't have much resources for any other kind. Design changes were made based on feedback.
